# Project Helix: Local-First Agentic Development Fleet

## 1. Project Vision & Elevator Pitch
Build an open-source desktop application that orchestrates a team of specialized, local-first AI agents. The user provides a high-level goal (e.g., "Build a todo app with user authentication"), and the fleet autonomously manages planning, coding, testing, and documentation. The system is a private, local alternative to cloud-based tools like Claude Code, offering superior specialization and zero data egress.

## 2. Core Value Proposition (Why this will succeed)
- **100% Local-First & Private**: All processing happens on the user's machine. Supports both cloud API keys (for power users) and local models via Ollama (for privacy).
- **True Specialization**: Unlike generalist assistants, this fleet features 7+ distinct agent roles (Architect, Coder, Reviewer, Debugger, QA, Researcher, Scribe) that collaborate like a real dev team.
- **Safety by Design**: Every agent action runs inside isolated Docker sandboxes to prevent system damage.
- **Open Source Ecosystem**: Targets the massive gap for a *desktop application* orchestrating local agent teams (alternatives are mostly CLI frameworks).

## 3. The Agent Fleet (Detailed Personas)

| Agent Name | Role Description | Primary Responsibilities |
| :--- | :--- | :--- |
| **Architect** | Project Planner | Breaks high-level goal into a structured `plan.md` and a `tasks.json` file. Defines tech stack requirements. |
| **Researcher** | Information Gatherer | Performs web searches (via Tavily/Serper API or local SearXNG) to provide up-to-date context on libraries, APIs, or errors. |
| **Principal Engineer** | Primary Coder | Writes and modifies code files. Uses tools: `read_file`, `write_file`, `execute_command` (sandboxed). Delegates to Debugger. |
| **Code Surgeon** | Debugger | Fixes specific bugs identified by QA or the Engineer. Focuses on surgical changes rather than full rewrites. |
| **Gatekeeper** | Code Reviewer | Reviews diffs for security flaws, performance issues, and style consistency. Outputs a `review.md` with actionable feedback. |
| **Guardian** | QA & Test Engineer | Writes unit/integration tests. Executes the application and attempts to "break" it (exploratory testing). |
| **Scribe** | Documentation Writer | Updates `README.md`, creates API docs, and writes user guides based on the final codebase. |

## 4. Technical Architecture Blueprint

### 4.1 Technology Stack
- **Backend / Orchestrator**: Python 3.11+ (FastAPI for internal API, if needed for UI bridge).
- **Orchestration Framework**: **CrewAI** (for initial MVP) or **AutoGen**. *Note: We will build a custom lightweight wrapper to allow fine-grained control, but CrewAI provides a robust starting point for agent memory and task delegation.*
- **Desktop UI**: **Tauri** (Rust backend for performance + HTML/CSS/JS frontend). Alternative quick-start: **Streamlit** for UI prototype.
- **Sandboxing**: **Docker Python SDK**. Every agent execution will launch a container with the project directory mounted as a volume.
- **Vector Memory**: **ChromaDB** or **sqlite-vec** (local, lightweight vector store for project context and long-term memory).
- **Model Interface**: **LiteLLM** (Unified interface for OpenAI, Anthropic, and Ollama local models).

### 4.2 System Flow
1. **Input**: User types goal in UI: *"Create a FastAPI endpoint for image uploads with S3 storage."*
2. **Orchestrator (Lead Agent)**:
   - Spawns **Architect**.
   - Architect generates `plan.md` and `tasks.json`.
3. **Execution Loop**:
   - For each task in `tasks.json`:
     - Lead Agent delegates to appropriate specialist (e.g., Coder for implementation).
     - Specialist executes actions inside a **Docker Sandbox**.
     - If error occurs, lead agent routes to **Debugger**.
     - If task complete, lead agent routes to **Reviewer**.
4. **Finalization**: **QA Agent** runs tests. **Scribe** updates docs.
5. **Output**: User receives a finished codebase in the project folder.

| :--- | :--- | :--- | :--- |

| **Model Support** | **Cloud API + Local Ollama** | Cloud Only | Cloud/Local (config required) |
| **Safety** | **Per-Agent Docker Sandbox** | Permission Prompts | No enforced sandbox by default |
| **Agent Specialization** | **7+ Hardcoded Roles** | Subagents (dynamic) | Customizable but generic |

## 6. Development Phases (MVP to v1.0)

### Phase 1: The Command Line Core (Current Sprint)
**Goal**: Make the agent fleet work from a terminal with a single command: `helix run "Build a Flask Hello World"`

**Implementation Plan for Phase 1:**
1. **Setup**: Create Python project with `crewai` or `autogen` dependency.
2. **Define Agents**: Create Python classes for **Architect**, **Coder**, **Researcher** (start with these 3).
3. **Integrate LiteLLM**: Ensure the Coder can use either `gpt-4o` (API) or `ollama/llama3` (local).
4. **Implement Docker Sandbox**:
   - Use `docker-py` to create a container with the project directory mounted.
   - Execute `write_file` and `execute_command` tools *inside* the container.
5. **Basic CLI**: Parse user input, run the fleet, output result to `./output/`.

### Phase 2: UI & Specialization
- Add **Debugger** and **Reviewer** agents.
- Build a **Streamlit UI** (quickest path to visual dashboard) or start **Tauri** frontend.
- Implement project memory with ChromaDB.

### Phase 3: Polishing & Community Launch
- Add **QA** and **Scribe** agents.
- Build one-click installer for Mac/Windows/Linux.
- Write comprehensive documentation and launch on GitHub.
